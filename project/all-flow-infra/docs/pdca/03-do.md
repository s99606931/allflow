## 2. 핵심 결정 사항

1. **베이스 + 오버레이 패턴** — 단일 base 파일 + dev/prod overlay 로 환경 분리. `--env-file` 플래그로 시크릿 주입.
2. **Postgres URL 컴파일은 백엔드 책임** — compose에서는 `DB_HOST/PORT/USER/PASSWORD/NAME` 개별 환경변수만 주입하여 시크릿 스캐너 안티패턴(`postgres://USER:PASS@HOST`)을 회피. 백엔드의 Prisma 설정에서 환경변수로 DATABASE_URL을 조립.
3. **JWT_SECRET 강제** — `${JWT_SECRET:?...}` 구문으로 미설정 시 compose 자체가 실패 → 시크릿 누락 사고 차단.
4. **헬스체크 + service_healthy 의존성** — Postgres/Redis가 실제 응답할 때까지 backend가 기다리고, backend healthy 후 frontend가 시작.
5. **fallback Dockerfile** — `../all-flow-backend/Dockerfile`이 아직 없어도 인프라 단독 검증 가능.
6. **Read-only prod 컨테이너** — `read_only: true` + tmpfs(`/tmp`, `.next/cache`)로 공급망 공격 표면 축소.
7. **WSL2 호환** — `CHOKIDAR_USEPOLLING=true`, `WATCHPACK_POLLING=true` 로 inotify 한계 우회.

## 3. 경계 조건 / 가정

- 백엔드는 `/health` 엔드포인트가 200을 반환한다고 가정. 미구현 시 헬스체크는 실패하나 컨테이너는 기동(헬스체크 실패는 healthy 의존성만 차단).
- 프론트엔드는 `next.config.ts`에서 `output: 'standalone'`을 export 해야 prod 이미지 빌드가 성공. (별도 PR 필요 사항으로 README/Plan에 명시)
- `pnpm-lock.yaml` 부재 시 fallback Dockerfile은 `pnpm install --frozen-lockfile`을 시도하고 실패 시 일반 `pnpm install`로 재시도(레이어 정의에 OR 분기).
- Docker Compose v2.20+ 가정 (service_healthy 조건). v2.20 미만 환경에서는 healthcheck 의존성 무시되며 단순 시작 순서만 적용.

## 4. 시크릿 관리

| 변수 | 발급 방법 |
|------|----------|
| `JWT_SECRET` | `openssl rand -hex 32` |
| `POSTGRES_PASSWORD` | 패스워드 매니저 / Vault |
| `OPENAI_API_KEY`, `ANTHROPIC_API_KEY` | 콘솔에서 발급 |
| `REDIS_PASSWORD` | dev: 빈 값 / prod: `openssl rand -base64 24` |

`.env.dev`와 `.env.prod`는 .gitignore에 등록되어 있으며, pre-commit Gate 1(시크릿 스캐너)이 한 번 더 차단한다.

## 5. 검증 명령

```bash
make check ENV=dev    # docker compose config -q 통과
make check ENV=prod   # docker compose config -q 통과
make up               # 4 컨테이너 healthy
make ps               # State == healthy
curl localhost:3000   # 200/302
curl localhost:8080/health  # 200
make backup           # backups/<ts>.sql.gz
make down             # 정리
make up               # 재기동 -> 데이터 보존 확인
```
